# Git 规范

本文件定义当前仓库的 Git 提交流程约束。任何违反本文件的操作都应视为流程违规，不得以效率为由跳过。

## 分支规则
- 禁止直接在 `master` 分支上进行编码、提交或其他变更操作
- 开始编码前必须先检查当前分支
- 若发现当前分支为 `master`，必须立即停止，不得继续修改文件
- 只有切换到正确的工作分支后，才允许继续开发

## 提交前检查
- 每次 `commit` 前必须执行编译检查
- 编译失败时禁止提交
- 只有在当前变更处于可编译状态时，才允许创建提交

## Push 规则
- 禁止自动执行 `push`
- `push` 必须由用户主动触发
- 在用户未明确要求前，只允许本地整理、检查和提交，不得上传远端
- 保留用户审查代码和提交内容的机会，不得绕过该审查窗口

## Commit Message 规则
- 提交信息必须使用以下格式：`<type>(<scope>): <中文简述>`
- `<type>` 必须从以下类型中选择：`feat`、`fix`、`docs`、`refactor`、`test`、`chore`、`perf`、`build`、`ci`、`revert`、`style`
- `feat` 用于新增功能
- `fix` 用于修复缺陷
- `docs` 用于文档更新
- `refactor` 用于不改变外部行为的重构
- `test` 用于测试补充或测试调整
- `chore` 用于杂项维护，不涉及业务功能变更
- `perf` 用于性能优化
- `build` 用于构建系统、依赖或打包配置调整
- `ci` 用于持续集成或自动化流程调整
- `revert` 用于回滚历史提交
- `style` 用于不影响逻辑的格式、排版或静态样式调整
- `<scope>` 应优先使用对应的 `change` 名称、模块名或目录名，要求稳定、可检索，不得随意缩写
- 若当前变更存在 `docs/changes/<change-name>/` 记录，`<scope>` 必须与 `<change-name>` 保持一致
- `<中文简述>` 必须直接说明本次提交的核心动作，使用明确动宾短语，不得写成无信息量描述
- 禁止使用非标准类型，例如 `mod`、`doc`、`update`、`misc`
- 禁止使用无信息量提交信息，例如 `fix bug`、`update code`、`test`
- 一次提交只允许表达一个核心意图；若包含多个独立意图，必须拆分为多个提交
- `WIP`、`temp`、`fixup`、`squash` 仅允许出现在本地临时整理中，禁止进入共享历史

## 推荐示例
- `feat(agent-memory): 新增会话摘要写入逻辑`
- `fix(workflow-branching): 修复条件分支未命中时的兜底路径`
- `docs(git-rule-standardize): 补充提交类型与 scope 约束`
- `refactor(tool-registry): 拆分工具注册与导出逻辑`

## 提交粒度规则
- 一个 commit 只解决一个明确问题，不得混入无关修改
- 功能变更、重构、格式化、文档补充应尽量分开提交
- 提交前必须检查暂存区内容，确保提交内容与提交信息一致

## 历史改写规则
- 允许在未共享的本地提交上执行 `commit --amend` 或交互式整理
- 已经 `push` 到共享分支的提交，禁止无说明地改写历史
- 未经用户明确允许，禁止执行 `push --force`

## 阻断条件
- 当前分支是 `master`
- 当前变更未通过编译检查
- 未经用户明确触发即执行 `push`
- 提交信息不符合规定格式
- 提交类型不在允许列表中
- 提交内容与提交说明明显不一致

## 执行原则
- Git 流程规范优先于个人习惯
- 发现违规条件时，应先停止操作并修正流程，再继续开发
